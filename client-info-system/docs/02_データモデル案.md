# 利用者情報整理システム データモデル案

## 概要

利用者を中心に、基本情報・支援情報・支援パターン（流れ）を関連付ける形で設計しています。  
実装時は「まずは1テーブルで全項目」から始め、必要に応じてテーブル分割する方法も可能です。

---

## エンティティ一覧

```
[利用者] 1 ---- N [支援パターン（流れ）]
   |
   +-- 基本情報（住所・移動・意思疎通・食事・排泄など）は利用者に直結
   +-- 支援についての情報（金銭・家族要望・好きなこと）も利用者に直結
```

---

## 1. 利用者（clients）

1利用者1レコード。基本情報・支援情報の多くをこのテーブルで保持します。

| カラム名（案） | 型 | 説明 |
|----------------|-----|------|
| id | UUID / 連番 | 主キー |
| name | 文字列 | 利用者氏名（表示用・検索用） |
| name_kana | 文字列 | よみがな（任意・検索用） |
| photo_url | 文字列 | 顔写真のURL（またはファイルパス） |
| gender | 文字列 / 選択 | 性別 |
| created_at | 日時 | 登録日時 |
| updated_at | 日時 | 最終更新日時 |
| memo | テキスト | その他メモ |

### 住所関連（同じテーブル内のカラム案）

| カラム名 | 型 | 説明 |
|----------|-----|------|
| address_home | 文字列 | 自宅住所 |
| address_home_map_url | 文字列 | 自宅のGoogleマップURL |
| address_after_school | 文字列 | 放課後等デイ（あれば） |
| address_after_school_map_url | 文字列 | 上記マップURL |
| address_tsusho | 文字列 | 通所（あれば） |
| address_tsusho_map_url | 文字列 | 上記マップURL |
| address_school | 文字列 | 学校（あれば） |
| address_school_map_url | 文字列 | 上記マップURL |
| address_other_places | テキスト | その他よく行く場所（複数可・JSONまたは改行区切り） |
| address_other_map_urls | テキスト | 上記のマップURL（対応する並びで保持） |

※「その他」が複数ある場合は、別テーブル「利用者・住所（複数）」にすると拡張しやすいです。

## 2. 支援パターン（support_patterns）※支援の流れ

「自宅→放デイ」「学校送迎」「外出〇〇」など、パターンごとに1レコード。

| カラム名 | 型 | 説明 |
|----------|-----|------|
| id | UUID / 連番 | 主キー |
| client_id | 外部キー | 利用者ID |
| name | 文字列 | パターン名（例：「自宅→放デイ→自宅」） |
| start_place | 文字列 | 支援スタート場所 |
| start_place_map_url | 文字列 | スタート場所のマップURL |
| end_place | 文字列 | 支援終了場所 |
| end_place_map_url | 文字列 | 終了場所のマップURL |
| timeline_schedule | テキスト | タイムスケジュール（時系列の説明） |
| flow_steps | テキスト / JSON | 支援の流れ（時系列のやること・注意点） |
| money_notes | テキスト | 金銭まわり注意点 |
| toileting_notes | テキスト | 排泄まわり注意点 |
| recommended_destinations | テキスト | 外出時：行き先のおすすめ・よく行く場所 |
| trouble_shooting | テキスト | 問題発生時の対処方法 |
| created_at | 日時 | 登録日時 |
| updated_at | 日時 | 最終更新日時 |

### 写真の扱い

- **案A**: 各パターンに「写真URLを複数」持たせる（photo_urls をJSONまたは別テーブル）
- **案B**: 「支援パターン・写真」の子テーブルで、1パターンに複数写真を紐付け、キャプションも保存

---

## 4. 運用・拡張のメモ

- **検索**: 利用者名・よみがな・注意点のキーワードで検索できると便利。全文検索またはタグを検討。
- **履歴**: 「誰がいつどの項目を変更したか」が必要なら、更新履歴テーブルを追加。
- **権限**: 将来的に「管理者のみ編集」「一般は閲覧のみ」などロールを分ける場合は、users テーブルと permissions を用意。
- **AI**: 自由記述テキストを要約して項目に振り分ける機能を後から追加する場合、このモデルのままで項目へマッピングしやすい形になっています。
